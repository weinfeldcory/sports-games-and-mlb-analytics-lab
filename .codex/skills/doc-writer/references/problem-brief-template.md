# Problem Brief Template

This reference mirrors the structure of the Google Doc template:

`https://docs.google.com/document/d/1RRlmLmSUKcmvOjxxvVvnreWBStUdBCFrB4bD7yhxEKc/edit`

## Canonical Outline

```md
Problem Brief Title

Date:
Author:
Supporting Documentation: X | Y | Z

Background

<brief summary of the work that led here>

Problem Statement

<specific description of the problem being addressed>

Method/Approach

<how the work will address the problem>

Defining Success

<metrics, criteria, and observable outcomes>
```

## Section Guidance

## Styling Notes

These are the main formatting details observed in the Google Docs template:

- title uses `Proxima Nova`, bold, `16pt`
- metadata lines use `Proxima Nova`
- paragraph line spacing is approximately `115`
- a divider appears between metadata and the brief body
- the body is a single-column table with `8` rows:
  - row 1: `Background` header
  - row 2: `Background` body
  - row 3: `Problem Statement` header
  - row 4: `Problem Statement` body
  - row 5: `Method/Approach` header
  - row 6: `Method/Approach` body
  - row 7: `Defining Success` header
  - row 8: `Defining Success` body
- header rows use:
  - teal background
  - bold white text
  - small cell padding
- body rows use:
  - white background
  - normal dark text

### Title

Use a direct title. Prefer the problem or initiative name over a slogan.

### Date

Use the document date the user wants. If not specified, use the current date.

### Author

Use the author name the user provides. If unclear, ask or leave as a placeholder only when necessary.

### Supporting Documentation

List the most relevant references only. Use a compact inline format such as:

`Supporting Documentation: PRD | Research notes | Dashboard`

### Background

Summarize the work that led to the brief:

- prior analysis
- product context
- observed issues
- why this brief exists now

Keep it informative but brief.

### Problem Statement

State the actual problem, not the proposed solution.

Good problem statements are:

- specific
- grounded in evidence or direct observation
- clear about who is affected
- clear about what is not working

### Method/Approach

Describe the proposed way to address the problem.

This can include:

- product changes
- operational changes
- research or experimentation
- documentation or process work

Keep it actionable and realistic.

### Defining Success

List the criteria that would show the work solved the problem.

Prefer:

- measurable outcomes
- decision criteria
- observable improvements
- explicit adoption or quality signals
